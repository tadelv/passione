# Contributing to Passione

Thanks for your interest in improving Passione — a modern web interface for the DE1 espresso machine via [Streamline-Bridge](https://github.com/tadelv/reaprime).

## Getting started

```bash
npm install
npm run dev          # Vite dev server with HMR + API/WS proxy
```

You need a Streamline-Bridge instance (or the mock server) to talk to. Point the dev proxy at it via `.env.local`:

```
VITE_GATEWAY_URL=http://<tablet-ip>:8080
VITE_WS_URL=ws://<tablet-ip>:8080
```

App code uses relative paths (`/api/v1/...`, `/ws/v1/...`); the Vite proxy routes them to the gateway. Default is `localhost:8080`.

### Prerequisites

- Node.js 20+
- A Streamline-Bridge instance, or the mock server for testing

## Before you start work

- **File or claim an issue first** for anything beyond a trivial fix. Issues live as GitHub Issues in [`tadelv/passione`](https://github.com/tadelv/passione/issues). Discuss the approach there before writing code on larger changes — it saves rework.
- **Read `CLAUDE.md`.** It documents the architecture, the key composables, and a "Gotchas" section of traps discovered the hard way. The boot-quiet rule, the shot normalizer, and the espresso→idle hook placement are all things you can easily get wrong without reading it first.

## Development workflow

1. Branch off `main`.
2. Make focused, surgical changes. One concern per branch.
3. Run the test suite (see below).
4. Open a PR against `main`.

### Commands

```bash
npm run dev          # dev server
npm run build        # static output in dist/
npm run preview      # preview the production build
npm run test:e2e     # Playwright end-to-end tests (against mock server)
```

## Code conventions

- **Vue 3 Composition API with `<script setup>`.** Match the existing style in `src/`.
- **Shared behavior lives in composables** (`src/composables/`), not duplicated per-page. In particular, shot data goes through `useShotNormalize` / `useChartNormalize` — do not roll your own normalization in a page or component.
- **No native-dialog-backed inputs** (`type="date"`, `type="time"`, `type="color"`, etc.). The Android Flutter host reloads the WebView when an Android native dialog dismisses. Use plain text inputs with `inputmode` + `pattern` hints.
- **Never use timers as guards or workarounds.** Use event-based flags and conditions. Timers are only for genuinely periodic tasks (polling, animation, heartbeats).
- **Boot-quiet on cold start.** Anything not user-blocking on first render must `await bootReady()` before touching the network — Wi-Fi and BLE share a radio on the target host, and a parallel handshake burst at boot drops the machine pairing. See the boot-quiet section in `CLAUDE.md`.
- **Don't over-engineer.** Add only what the task requires — no speculative abstractions, no backwards-compat shims, no comments that restate what the code already says.

### Design principles

Passione aims for a warm, confident, Apple-Home-style dark UI. Before adding UI, skim the Design Principles in `CLAUDE.md` (and `.impeccable.md` for the full design system). Touch targets are 44–56px minimum; accessibility targets WCAG AAA where practical.

## Commit messages

Use [Conventional Commits](https://www.conventionalcommits.org/) with a scope:

```
feat(recipe-editor): add per-grinder RPM clamp
fix(history-chart): trim to shot range
docs(readme): document the screensaver modes
perf(boot): quiet the cold-start burst
```

Common types: `feat`, `fix`, `docs`, `perf`, `refactor`, `test`, `chore`. Keep the subject under ~50 chars; add a body when the *why* isn't obvious from the subject.

## Pull requests

- Keep the description tight: **what** changed and **why**. Link the issue it closes.
- Include a short test plan only if it's non-obvious.
- Make sure `npm run build` succeeds and `npm run test:e2e` passes.
- For UI changes, verify the behavior in a browser — type checks and tests confirm correctness, not feel.

Note: `docs/screenshots/*.png` are regenerated by the e2e screenshot test and will show as modified after any e2e run. Exclude them from unrelated commits; regenerate intentionally when you want fresh shots.

## Reporting bugs and requesting features

Open a [GitHub issue](https://github.com/tadelv/passione/issues). For bugs, include:

- What you did, what you expected, what happened.
- Streamline-Bridge version and target host (tablet / desktop / mock).
- Console errors or relevant snapshot data if available.
